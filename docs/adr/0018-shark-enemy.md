# ADR 0018 - Shark Enemy: Spritesheet Assembly, Fight Tier, and Deep-Water Spawn

**Date:** 2026-06-16
**Status:** Accepted

## Context

Shark fills a hard-tier slot in the fish enemy roster, positioned alongside HammerHeadShark in difficulty. The Shark_1 asset pack ships as individual PNGs across per-animation folders with two distinct native sizes (move: 1036x483 px, die: 1060x512 px), requiring a PIL stitching step and a centering decision before use. Several decisions arose regarding spritesheet layout, canonical cell size, fight tier, display size, drift speed, spawn zone, and the attack animation row.

## Decisions

### 1. `Shark` extends `CatchableFish` via the `spriteFrameSize` pattern

`Shark` follows the same two-layer template-method hierarchy as `LionFish`, `Tuna`, `HammerHeadShark`, `SwordFish`, and `ClownFish`: it extends `CatchableFish` and overrides `draw()` and `_drawCapturedSprite()` with separate `_sw`/`_sh` fields sourced from `spriteFrameSize`. This decouples source-frame stride (1060x512 px canonical cell) from display size (530x256 px) without modifying the base class.

All capture glow, blink, arc, and shrink animations are inherited from `EnemyWithAnimation.drawCaptured()` for free via the `drawCaptured()` -> `_drawCapturedSprite()` template method.

### 2. Canonical cell is 1060x512 px (die-frame natural size); move frames centred inside

The Shark_1 pack provides two used animation groups with different natural sizes:
- `Shark_move_1`: 10 frames at 1036x483 px
- `Shark_die_1`: 10 frames at 1060x512 px

Using the larger die-frame dimensions as the canonical cell avoids distorting the captured-pose sprite and keeps the `frameX * _sw` stride formula universal across both rows. Move frames are centred inside the cell with a transparent surround:

```
offset_x = (1060 - 1036) // 2 = 12 px
offset_y = (512 - 483) // 2   = 14 px
```

Final spritesheet: **10600x1024 px** (10 cols x 1060 px, 2 rows x 512 px).

### 3. 2-row layout: Row 0 = Shark_move_1 (10 frames), Row 1 = Shark_die_1 (10 frames); `dieFrameY = 1`

`maxFrameY = 1` in the EnemyFactory spec ensures only row 0 cycles during swim. Row 1 is accessed exclusively via `dieFrameY = 1`, consistent with `LionFish`, `HammerHeadShark`, `SwordFish`, `Tuna`, and `ClownFish`. The `_drawCapturedSprite()` override accesses the die row directly (`dieFrameX * _sw, dieFrameY * _sh`) without going through the frame animation cycle.

### 4. Fight spec tier: `strength = 60, escape_rate = 2.0` (hard tier)

| Fish | Strength | Escape Rate |
|------|----------|-------------|
| ClownFish | 5 | 1.2 |
| ButterflyFish | 5 | 1.5 |
| LionFish | 15 | 2.5 |
| Octopus | 20 | 1.8 |
| Crab | 40 | 2.2 |
| HammerHeadShark | 50 | 2.0 |
| SwordFish | 50 | 2.0 |
| **Shark** | **60** | **2.0** |
| Tuna | 60 | 3.0 |

Shark sits in the hard tier above HammerHeadShark (strength 50) and below Tuna (escape_rate 3.0). The matching escape_rate with HammerHeadShark (2.0) makes Shark harder to land purely through the strength difference, with no increase in escape speed.

### 5. Display size: 530x256 px (50% of canonical cell)

The canonical cell (1060x512 px) is downscaled to 530x256 px (50%) for display. This gives Shark a large, imposing on-screen size befitting a hard-tier enemy while avoiding excessive canvas coverage.

### 6. Drift speed: `SHARK_DRIFT_SPEED = 4.0` px/tick (same tier as Tuna and Crab)

Shark drifts at 4.0 px/tick, matching Tuna and Crab. This places it above HammerHeadShark (3.5) and below SwordFish (4.5), making it fast but not the fastest enemy in the roster.

### 7. Deep-water spawn zone: `WATER_SURFACE_Y + 200` to `canvasHeight - fishHeight`

Shark spawns in the deepest tier, 100 px below HammerHeadShark (WATER_SURFACE_Y + 100). This reinforces its identity as a lurking deep-water predator and requires players to cast further to encounter it.

### 8. `Shark_attack_1` animation not used (deferred)

The Shark_1 asset pack includes a `Shark_attack_1` folder (10 frames at 1075x529 px). It is not wired in this release:

- A third animation row would require new state-machine logic in `EnemyWithAnimation` to trigger and exit the attack state, which is out of scope for a basic `CatchableFish` addition.
- Direction reversal is handled by canvas `ctx.scale(-1, 1)` on the left-facing sprite, matching every other directional fish.
- The attack row can be added in a future ADR when the `EnemyWithAnimation` state machine is extended.

## Consequences

- A new `Shark` enemy class spawns in every game session, swimming at depth and requiring sustained button-mashing to land.
- `images/fishes/shark_sprite.png` (10600x1024 px) is generated by `scripts/assemble_shark_sprite.py` and must be regenerated if source PNGs change.
- The attack animation row remains on disk but is not loaded; future attack support requires a new ADR and state machine changes.
