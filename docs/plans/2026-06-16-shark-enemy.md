## Summary

**Goal:** Add a `Shark` enemy fish (extends `CatchableFish`) with spec `{ strength: 60, escape_rate: 2.0 }`, assembled from 30 individual PNG sprite frames (move×10 + die×10; attack row deferred) into `images/fishes/shark_sprite.png`, registered in `EnemyFactory`, declared `display: none` in `main.css`, spawned once in `Game.js`, covered by unit tests, and documented in ADR-0018.

## Out of Scope

- Attack animation row (10 frames at 1075×529) — not used; the 2-row move+die layout matches every other CatchableFish.
- Score value for Shark — `ScoreSystem` is unchanged; the fight spec is the only game-balance lever asked for.

## Approach

**Chosen:** Mirror `HammerHeadShark` in every detail — same class shape, same EnemyFactory spec/branch, same 2-row spritesheet layout, same test suite structure.
**Why:** Every CatchableFish follows this exact pattern; deviating adds maintenance cost with no player-visible benefit.

## Context for Implementer

Shark frames vary in natural size (move: 1036×483, die: 1060×512). The **canonical cell** is the die-frame size — 1060×512 — because the die row is pasted at natural size and the cell must accommodate it. Move frames are centered inside the canonical cell (offset x=12, y=14). This is identical to the clown_fish assembly in `scripts/assemble_clown_fish_sprite.py`.

The `SPRITE_SWIM_MAX_FRAME_Y = 1` constant (EnemyFactory.js:14) is the correct `maxFrameY` for all CatchableFish; only row 0 (swim) is cycled, row 1 (die) is accessed directly via `dieFrameY`.

`FISH_SPECS['shark']` is already present in `src/constants.js:96` — do **not** add a duplicate entry.
